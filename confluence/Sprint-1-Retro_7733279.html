Team 55 IT Project : Sprint 1 Retro

Created by Kian DSOUZA, last modified by Ian Chen on Sep 28, 2022

02 Sep 2022

During Sprint 1, on a broad level, the team discussed:

  1. What went well in the iteration

    Excellent Communication and Integration Tools - Yi Sheon Tan
    We were able to set up a very comprehensive Slack channel relatively quickly, that is integrated with Confluence, Jira and GitHub, with bots that notifies everyone when an issue status changes, and when a pull request is opened etc.

    Code Standards and Improvement - Jack Woodman
    The team spent time reviewing code standards and programming styles with the use of linting, and improved as we went on. We also revised already written code as we progressed to find better practices.

    Lots of time spent learning learning new software tools - Kian DSOUZA
    After deciding the stack, the team spent time getting hands on learning on the tools using online docs

    Great discussion on core functions - Calvin
    We had some good discussions about the implementation of core functions. E.g for the search function, we initially had a complicated design that sorts photos by date, after a thorough discussion, we decided to do a simpler approach and the code is simplified.

    Defining requirements clearly - Ian Chen
    It was good to see that everyone was quick to work with the client to come up with the most pertinent requirements given the time restrictions.

  2. What could be improved

    Standardising coding conventions and use of libraries - Yi Sheon Tan
    In the first sprint, several conflicts arose regarding the use of certain libraries, coding style and conventions. We need to standardise approaches, and make sure everyone is aware of these standards to prevent wasted time and effort.

    Team involvement in decision making - Kian DSOUZA
    The entire team as a whole can show more involvement in decision making, and keeping up to date with the stack and tools to be used.

    Consolidation of Functions and Files - Jack Woodman
    The team can work on improving how code is used and reused throughout the project, in order to promote (appropriate) reuse and cull deprecated code to prevent bloat. This includes concreting or removing debugging functions that are used in production

    Better team forming process - Calvin
    We can be a more united team by exercising more team-forming activities such as having food together or doing chit-chat about each team member’s personal life.

    Being more open to autonomously researching and learning how to use libraries well - Ian Chen
    We can increase productively and our knowledge by making sure that we are implementing features in a reasonable efficient and maintainable manner, and to continuously question and critique the purpose of writing code in a particular way - e.g., is it imperative to extract this component if it’s only for this one part of the app?

  3. What will we commit to improve in the next iteration

    Discuss approach with team before implementing the code - Kian DSOUZA
    just so everyone we can get valuable input from the team.

    Better decision-making processes - Calvin
    We will exercise the new decision-making processes described in Team Decision Making just to get everyone involved in the project.

    Make Delegation Clear and Understood - Jack Woodman
    When delegating tasks to others (as per leaders setup), make sure tasks are well understood and appropriately shared

    Codify version control (git)procedures - Ian Chen
    Write up and distribute our conventions for using git (rebase only, linear commit history, no merge commits) to increase productivity and level-up everyone’s git skills. Now in Git Repo Standards.

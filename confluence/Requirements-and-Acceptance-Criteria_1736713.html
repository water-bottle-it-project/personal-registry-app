 Combining Requirements and Acceptance Criteria

We were able to combine requirements and acceptance criteria because our requirements and user stories have already been split up into smaller chunks. We now have 1 requirement = 1 user story = 1 acceptance criteria, so we have merged them to normalise our data, simplify linking of requirements, user stories, and acceptance criteria to test cases, and prevent duplication.

 Functional Requirements and their Acceptance Criteria

John is a user who wishes to catalogue memories and personal artifacts in the form of photos and detailed textual descriptions.

NICE TO HAVE items are not required, but merely a possible extension of the scope, time permitting.

Acceptance criteria: Given-When-Then structure.

Memories

M1

Create a memory and upload 0:n photos

John wants to create a new record of a memory or artifact, with 0 or many photos.

  • He doesn’t always want a description, but when he does, he wants to be able to provide ample detail.

  • He also wants to add photo captions and a manual date for photos if the date metadata is incorrect or missing.

  • He wants to add a memory to 0:n collections such as ‘family’, ‘friends’, ‘food’ or ‘travel’ since he is a fan of organising things by groups.

HIGH

LARGE

I have a title and a date range for the memory, and 0:n photos with optional caption, location and photo date.

I input the data into the ‘Create Memory’ inputs and click save.

I will see the memory be successfully created and stored in the list of memories, and see a visual notification confirming this

M2

Create - manually choose location

John may want to manually specify location info (landmark, address, or coordinates) to the database entry associated with the photo.

MEDIUM

LARGE

I have a location name I want to use.

I type this location name into the map search box

I can choose the correct suggestion, see the map update, and when I click save, see it persisted in the memory page.

M3

View a memory

John wants to revisit a past memory/artifact and easily browse the description and photos from that time.

For each photo, he wants to see the position of the photo e.g. 1/15, captions, location, and date if they are present, as well as any other metadata contained within the image e.g. aperture & device.

HIGH

MEDIUM

I have created at least one memory

I select a memory from the all memories view (with and without search query)

I will be presented with the view of the memory. This page will be persistent - the i.d. is in the URL, and reloading takes me back to the same memory.

M4

View all memories, sorted in chronological order

John wants to look back over a hectic month, year, or years and see a timeline of all memories - both newest and oldest first.

He should be able to see the title, description, its collections, the number of photos, and the preview of the first photo. Long titles and descriptions should be trimmed.

HIGH

SMALL

I have created multiple memories and am viewing them in the all memories view

I choose to sort the timeline by date, ascending or descending

I will be presented again with the timeline view, this time with the memories sorted by either ascending or descending order

M5

View all memories, paginated

John wants to have his memories displayed in pages so that he can jump between different pages quickly instead of doing an infinite scroll to find memories in the middle.

  • Page numbers should be persistent i.e. url has a query param ?page=2

  • Reloading Ctrl+r should take you back to the same page.

  • Must be able to press ← and → in the browser (or use Mouse4 and Mouse5 buttons) to go back and forward between pages.

  • Preserve Newest/Oldest ordering of memories

  • Invalid page numbers should be handled gracefully

    • ?page=2+3 (array) =>

    • ?page=abc where abc is some arbitrary non-number string =>

    • ?page=-1 (non-positive) =>

    • ?page=k where k > total number of pages => ?page=lastpage

HIGH

MEDIUM

I am on the all memories page and want to jump to a specific page

I press the page number at the bottom of the page or the first, prev, next, last buttons.

or I manually input the URL e.g. I have a browser bookmark.

The grid of memory cards updates with the memories from that page, while preserving the newest/oldest ordering of memories chosen. If the page number is invalid, then we are redirected following the dot point rules described in the user story

LOW

M15

Create collection inline when applying collections to memories

John doesn’t want to navigate from the current page while adding a collection label to a memory because this involves a context switch or losing his progress.

Oct 22 update: Decided to not implement (hence the nice-to-have priority). Users can instead open “Collections” in a new tab (right click or long-press → new tab), create the collection there, then switch back to the create/edit memory tab. Using TRPC + react-query, the new collection can be seen in the multi-select drop-down while keeping existing unsaved changes to the memory - no need to reload the edit/create page. Demoed in presentation.

Other functional reqs

F1

Toggle light/dark mode

John wants to be able to use the website in a variety of lighting conditions, and prefers dark mode when the environment is dark around him

  • Doesn’t need to sync across devices - it can be a browser-specific cookie

  • UI’s colours need to adapt to both light and dark modes well.

  • Should sync instantly across different tabs in the same browser

MEDIUM

SMALL

I want to switch to dark mode from any page

I press the toggle colour scheme icon button on the navbar, from any page

The colour scheme is toggled from light ↔︎ dark instantly, and across all tabs too.

F2

Protected pages - authentication, security, and privacy

John wants to have confidence in the security and privacy of the site, so all the pages with user-generated content can only be accessible when sign in. When he signs out, he expects any cached backend queries to be cleared to

HIGH

SMALL

I am not signed in.

I attempt to access /collections etc. when logged out, either by clicking in the navbar, or manually inputting the URL.

We are redirected to the sign in page, with an alert stating I must be authed, with a link to register if I am a new user.

Non-functional requirements

  • John wants the site to be responsive and performant. He may need to access it on an underpowered laptop, so the time to First Meaningful Paint needs to be low. Data fetching can occur later, but he wants to be able to tell that there is content being processed ASAP.

  • John wants the colours of the website to be reasonably tasteful and un-offensive. Given that photos and text content should take priority, the website’s background UI needs be relatively neutral.

  • John wants a consistent user interface (e.g. spacing, corner radius, colours, font type and size), that is easy to follow

    • For example, the spacing between search bar and the grid of memory/photo results below should be consistent for the all memories, photos, and collections pages.

    • Consistent indigo accent colour throughout the UI.
